F-SL2 · Spec

New sidebar IA · debut

  • Primary: Home · Start (the two universal operator anchors)
  • Speak FAB: separated · prominent · not a Primary peer
  • All: Sales · Projects · Schedule · Money · Team+Ops · Marketing (in sidebar order · democratic)
  • Sub-items: Clients nested under Team+Ops
  • Bottom: Settings · Audit

Locked Sales canon applied

  • Pipeline-first, proposal-enabled, relationship-aware, pre-deal only.
  • Pipeline is the primary lens · Proposals reached through deal detail (not a peer here)
  • Leads = stage inside Pipeline · not a separate sub-tier
  • Stage tags as quiet text labels · status colors reserved for state

Desktop density choices

  • 4-column row · name + stage tag | scope + meta | status chip + state + amount | arrow. Mobile collapsed these into one stacked block.
  • Right rail-led pipeline synthesis · RH card + funnel + Next-step queue. Main column stays a list.
  • Next-step queue in rail · 4 highest-impact actions across all deals. Click each to drill to deal detail with that action highlighted.
  • Funnel includes Won + Lost summary · context not detail. Won 4 deals last quarter; Lost 1 in last 30d.

Sort logic

  • Section 1: Needs attention (4) · urgency-sorted: red (stale/overdue) → amber (your reply) → green (ready to send)
  • Section 2: On track (2) · soonest next event first
  • Won + Lost deals collapse into funnel footer · not surfaced as rows here

Row anatomy

  • Name block: deal name + stage tag + meta line (EST id, dates, source)
  • Scope block: headline + sub-detail (address, scope hint)
  • State block: status chip + state line + amount (monospace)
  • Arrow: drill to F-SL4 Deal detail

What this surface refuses to be

  • Not a CRM workstation with custom fields
  • Not a kanban board with drag-and-drop stage columns
  • Not a Salesforce-style opportunity grid
  • Not a HubSpot deal pipeline view
  • Not a filter-and-sort dashboard
  • Not a separate "Proposals" peer view

Doctrine bound

  • One primary cognitive job — see pipeline state · drill where action lives
  • Smart summary leads — rail RH card opens the page
  • Sort by attention · not stage-grouped first · not alphabetical
  • Red is chip-tier · per feedback_red_is_chip_tier_not_row_tier.md
  • Trust surfaces exception-tier · no transcript/audit nav
  • Right Hand named once · no agent names elsewhere

Routes

  • + New estimate (breadcrumb action) → F-S1 Start sheet
  • Deal row click → F-SL4 Deal detail
  • Next-step queue item → deal detail with action highlighted
  • Sidebar Home → F-A2 Owner Home
  • Sidebar Start → F-S1 Start sheet (desktop variant when built)
  • Speak FAB → RH conversation
  • Other sidebar domains → respective destinations

What's deferred

  • Forecast view (weighted pipeline) — V1.5+
  • Lost-deal analysis surface — V1.5+
  • Per-deal activity timeline aggregation — V1.5+
  • Stage drag-and-drop (avoid kanban temptation) — never

F-SL2 Grid · Spec

Big-picture canon applied

  • Comparison view · every deal visible at once
  • 4-column desktop grid · no attention compression
  • Tile content disciplined: name + stage + state chip + deal value/rhythm · scan not detail
  • Map view (V1.5+) shown as third toggle option (deferred)
  • Drill into any tile → F-SL4 Deal Detail
What this surface proves The new sidebar IA reads correctly at desktop density. Primary becomes the operator's universal anchors (Home + Start), Speak separates as the modality button, and the 6 Layer A domains live democratically in All. Pipeline-first Sales is honest about the work — the rail carries the synthesis, the list carries the deals, and the Next-step queue surfaces the 4 highest-impact actions across the funnel.
What this surface refuses to be Not Salesforce. Not HubSpot. Not a kanban grid. Not a custom-field forest. F-SL2 is the pipeline synthesis with drill affordances underneath. Proposals don't get a peer view; they live inside deal detail because a proposal exists in service of a deal.
