F-B1b · Mobile Decision Card · Edit mode

The critical state. Operator taps Edit on the pending Decision Card (F-B1) and lands here. This is where trust either holds or fails: if editing feels opaque or like fighting Right Hand, the operator won't approve. If editing feels like Right Hand made a sensible first pass that I can adjust naturally — change a quantity, override a sub cost, tweak markup — trust holds. Quantities · labor hours · material costs · sub costs · markup all editable per line. Global markup % at top. Right Hand's suggestion stays visible as the operator overrides. Trust check re-runs on Save. Plain English throughout — no AltitudePacket / model_suggested / Policy Gate vocabulary in operator copy.

What's editable

  • Per line: Qty (stepper) · Labor hours (stepper) · Material cost · Sub cost · Markup % override
  • Global: Markup % (applies to all lines unless overridden) · Materials buffer %
  • Structural: + Add scope line · swipe-to-delete a line
  • Assumptions: tap a confirmed assumption → re-confirm or change
  • Scope wording: tap line name → text edit (the client-facing copy)

What's NOT editable on this surface

  • Change-order identity (CO-002 stays CO-002 · a different CO would be a new draft)
  • Trust check pass/fail itself (re-runs deterministically based on your inputs)
  • Audit substrate (immutable · your edits become events in audit)
  • Right Hand's suggestions (they remain visible · you override, but you don't delete RH's reasoning)

Right Hand's role during edit (plain English)

  • Each line shows RH's suggested total below the operator's current total
  • "✓ matches" when current = RH suggestion (line 1 here)
  • "$X · RH suggested $Y" when user has overridden
  • This is the operator-facing version of system_final_* vs model_suggested_* — canon vocabulary lives in audit substrate

Override-as-preference pattern

  • When operator overrides RH consistently (e.g., specific sub on specific scope category), Right Hand surfaces the override callout: "Save this as your preference so I use $X on future jobs?"
  • Saves to tenant pricing memory · feeds future cohort comparables
  • This is how RH learns the contractor without ever claiming to "learn" — preferences are saved explicitly with consent
  • Per kerf_scope_exclusions.md: AI learns the owner, owner does not train AI

Line-card warning state

  • Amber border when a line is "over typical" — flagged by pricing source-class bounds (V6/V7 in audit)
  • Surfaced as scannable hint, NOT blocker · operator can save anyway
  • Tap the warning to expand — shows what bound was exceeded in plain English ("Your typical paint cost on jobs this size is $1,200–$1,600. This line is $1,850.")
  • Re-checked on Save · trust check may flag for review but won't auto-reject

Voice editing — input modality, page is commit gate

  • 🎤 Speak supports edit instructions on this surface. Operator long-presses Speak in edit mode and says: "Change markup to 25%" or "Set Patel electrical sub cost to $2,800" or "Reduce painting line by 20%"
  • Right Hand interprets the voice intent and applies the edit to the visible state — the operator sees the change in the line cards / totals immediately
  • Voice can edit multiple lines in one capture ("Bump markup to 25% and add an outlet to the pantry — call it 2 hours and $200 material")
  • Voice is the input modality; the visual page is the commit gate. Voice never auto-saves — operator must visually review the applied edits and tap Save
  • This honors AI disclosure §13: "Review before approval" — voice doesn't bypass review
  • Voice ambiguity (e.g., "make the labor reasonable") triggers Clarification agent inline: "Right Hand: which line — pantry framing or shelving? And what's reasonable to you here?"
  • Visual edits and voice edits are equivalent in audit · both become edit events with the same provenance shape (operator override vs RH suggestion)

Save / Cancel / Discard semantics

  • Cancel (top-left) — exit edit mode, return to F-B1 pending state, no changes saved (confirmation modal if edits exist)
  • Save (top-right) — persist edits, trust check re-runs, return to F-B1 with updated totals + recheck-running state, then resolves to passed/failed
  • Discard edits (bottom) — clear all edits without exiting (operator can keep editing fresh)
  • Save button stays disabled until at least one edit is made · prevents no-op saves

Trust posture preserved

  • Original draft preserved · edits become an event in audit (per kerf_transcript_canon.md overlay pattern, applied to draft)
  • Audit substrate shows: "Operator edit · Line 4 sub cost · $2,800 → $2,500 · 14:52"
  • RH's original suggestions preserved · operator's overrides preserved · provenance chain intact
  • Trust check re-runs on Save · returns pass/warn/fail · operator sees plain-English result

Routes

  • Save → trust re-check → F-B1 (pending decision · updated totals)
