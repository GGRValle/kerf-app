F-BK1b Grid · Spec

4-question rule

  • Macro / comparison / detail? Comparison (peer unreconciled items at macro scan by state)
  • List / grid / both? List + Grid · Grid for state-based scan (sibling toggle to Frame 1)
  • Cognitive job? Scan 17 items by recon-state × amount · what needs which kind of operator touch
  • Hidden until drill? Per-transaction history · PO-to-invoice match · journal entry composition (drill to transaction detail)

Grid anatomy · 17 tiles

  • 4-column desktop grid · same 17 items as List frame (one dataset, two views)
  • gt-name = source/vendor · gt-phase = recon bucket + context · gt-chip = state · gt-rhythm = $
  • Red chip + amber `.grid-tile.attention` border = needs my hand (5 tiles)
  • Green chip + green `.grid-tile.success` border = auto-matched, ready to confirm (8 tiles)
  • Amber chip + no border = unmatched payment-in, needs ID work (4 tiles)
  • `.grid-tile.risk` class not used — Lane 3 doctrine: red is chip-tier only (CSS swept 2026-05-24)

Money model

  • Display: dollars · Store: integer cents on every amount
  • Lifecycle states: needs_judgment · auto_matched · unmatched · confirmed
  • NEVER auto-confirms · D-046 lock

Fillable function (memory canon)

  • Filled-by chip preserved on Grid rail (parity with List frame)
  • Per feedback_bookkeeping_is_a_fillable_function.md
What this surface enforcesBookkeeping as a workflow queue, not a ledger. Three buckets ordered by required attention type: judgment first, batch-confirm second, ID-work third. Filled-by chip surfaces the role doing the work — Owner today, Delegate available, External MCP a V1.5 option. Never auto-confirms.
What this surface refuses to beNot the general ledger. Not transaction detail. Not the QB import screen. F-BK1b triages unreconciled items by required attention — drill to transaction detail for line items · journal entry composition stays in QB until V1.5 bidirectional sync lands.
