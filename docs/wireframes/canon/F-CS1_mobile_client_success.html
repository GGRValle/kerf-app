F-CS1 · Mobile Client Success Detail · the success-lens projection of a client

Per-client deep view of post-close relationship state. Reached via Marketing → Success list, or from a project closeout follow-up prompt. Distinct from F-CL5/CL6 (the authoritative entity surface with all projects, AR, AP, files); this surface answers "are we due for an anniversary touch, a review ask, a referral request? Are they an advocate?" Drill to canonical record for the full client view.

Success-lens vs canonical record

F-CS1/CS2 is the success-lens projection of the client entity — focused narrowly on relationship / marketing / repeat-business state. F-CL5/CL6 (canonical record) carries the full ledger — all projects, all AR/AP, all comms, all files. Same underlying client, different lens. Operator clicks "Open canonical client record" in the Watching footer to drill from success view to full record.

Six operator questions answered above-the-fold

  • Where are they in the lifecycle? Header advocate chip + Lifecycle 5-step stepper
  • How fresh is the relationship? Touch cadence section (last · cadence · next)
  • Are they advocating? Advocacy state section (reviews · referrals · worth · status)
  • What action is open right now? Open marketing actions (3 cards with state chips)
  • How happy are they? NPS history with trend line
  • What's Right Hand watching? Watching card with 3 monitors + canonical-record drill

Grid doctrine applied

  • DETAILThis is a detail surface — single client success depth. No grid view. No view-mode toggle.
  • Single-frame wireframe per the 4-question dispatch
  • No tabs · Success is a focused single-lens projection (vs F-CL5's six lenses)

Disciplines applied

  • D1Business brain, not file cabinet — RH synthesis + 6 glanceable sections. Individual messages / full history sit one click deeper (canonical record).
  • D2Red is chip-tier only — no full-row red borders anywhere. State chips for "Ready" / "Queued" / "Scheduled" stay amber/cyan/muted. Red reserved for genuine risk (none on this advocate-stage client).
  • D3No agent names — "anniversary note drafted" / "ask queued" / "drafted by cadence engine" — name the work, not the agent. Right Hand named only on RH summary + Watching card.
  • D4Translate canon vocab — "advocate status," "top quintile," "your book," "stable-high" — plainspoken contractor English. No "lifecycle segment 4," "cohort routing envelope," "P25–P75."
  • D5Bottom bar default state — reached via Marketing → Success not via Clients sub-item; no bottom-bar tab active.

Six synthesis sections

  • Lifecycle stage · 5-step horizontal stepper (Lead · First job · Repeat · Advocate · Lapsed) · current advocate stage with context line
  • Touch cadence · last touch · cadence target + on-track state · next scheduled touch
  • Advocacy state · reviews left · referrals sent · referral attribution value · advocate status
  • Open marketing actions · 3 action cards (anniversary · review ask · holiday touch) with state chips
  • NPS history · per-project NPS rows + trend line (stable-high)
  • What I'm watching for you · RH footer card · 3 monitors + canonical-record drill

Section ordering rationale

Lifecycle stage leads because it answers "where are they?" most efficiently. Touch cadence is second because it sets up "what's open right now." Advocacy is third because it's the success-lens payoff (what makes this client worth tending). Open actions are fourth — what to actually do. NPS is fifth — confirmation the relationship is healthy. Watching closes with what's coming up + drill to canonical.
