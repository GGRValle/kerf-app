F-SL4 · Spec

Action-led desktop deal detail

  • Lead with state synthesis · header + RH sentence + 3 execution cards span ~32% of viewport before tabs
  • Primary action card · full-width, prominent, green-accent · between execution row and tab strip
  • Tabs as drill lenses · Overview · Activity · Proposal · Notes · Files
  • Slim rail · recent activity timeline + quiet Audit row only

Why action card full-width above tabs (vs F-PR4's rail-placement)

F-PR4 put CO-002 in the right rail because Project Detail is execution-led (state is the primary read). Deal Detail is action-led — the operator opens a deal to do something with it. The action gets visual prominence as a full-width card so the surface reads "here's the deal · here's what to do next." The 3-card execution row above gives state context; the action card asks for the move.

Execution row content

  • Activity · last touch: what was the most recent contact + total touch count
  • Proposal status: lifecycle state (drafted / sent / signed) + readiness signal
  • Money · deal posture: total $ + breakdown summary + deposit terms + margin posture

Locked Sales canon applied

  • Pipeline-first, proposal-enabled, relationship-aware, pre-deal only.
  • Proposal exists in service of a deal · "View proposal" routes to F-PV1 client view
  • Client = contracting counterparty · Patel-direct: Anjali is client
  • Relationships pair surfaces both contracting client and referral source · canonical records live in Team+Ops > Clients

Routes

  • Send proposal CTA → V18 gate (external send) · post-send routes to Proposal tab with sent state
  • Preview as client → F-PV1 Proposal Preview
  • "View proposal →" inline action in Pricing block → F-PV1
  • Edit proposal (header action) → F-G1 Draft Review
  • Log a touch (header action) → activity capture sheet
  • Anjali rel-block → Team+Ops > Clients > Anjali Patel
  • Cassandra rel-block → Team+Ops > Clients > Cassandra Reyes
  • Tab switching → Activity / Proposal / Notes / Files lenses
  • Audit deep-link → F-H1 with this deal's event chain filtered

What this surface refuses to be

  • Not a Salesforce-style opportunity record
  • Not a HubSpot deal detail
  • Not 5 mini-dashboards stitched together
  • Not a forensics console
  • Not a "edit every field" intake form

What this surface proves Sales is action-led, not state-led. The execution row gives state context (recent activity · proposal status · money posture) but the surface gravity sits on the primary action card. "Here's the deal · here's what to do next" reads at first glance without drilling into tabs. The slim rail (activity timeline + quiet Audit) supports the surface without competing with it.
What this surface refuses to be Not Salesforce. Not HubSpot. Not a forensics console with 12 panels. Not a "click to edit 47 fields" detail page. F-SL4 is the deal lens with one prominent next-action card and tabbed drill underneath. Proposal lives inside the deal because a proposal exists in service of a deal.
