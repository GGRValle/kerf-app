F-BK1a Grid · Spec

4-question rule

  • Macro / comparison / detail? Comparison (state × amount scan across 17 unreconciled items)
  • List / grid / both? Grid (sibling toggle to Frame 1 List)
  • Cognitive job? Which judgment-needed items are biggest? · which auto-matches can I batch-confirm?
  • Hidden until drill? Transaction detail · suggested match reasoning · journal entry (tile click → row detail)

Tile anatomy

  • 2-column mobile grid · 17 tiles · same 17 items as List frame
  • gt-name = source/vendor · gt-phase = recon bucket + context · gt-chip = state · gt-rhythm = $
  • Red chip + amber `.grid-tile.attention` border = needs judgment (5 tiles)
  • Green chip + green `.grid-tile.success` border = auto-matched, ready to confirm (8 tiles)
  • Amber chip = unmatched payment-in, needs ID work (4 tiles)
  • `.grid-tile.risk` class not used — Lane 3 doctrine: red is chip-tier only
What this surface enforcesBookkeeping as a workflow queue, not a ledger. Three buckets order by required attention type: judgment first, batch-confirm second, ID-work third. Filled-by chip surfaces the role doing the work — Owner today, Delegate available, External MCP a V1.5 option. Never auto-confirms.
What this surface refuses to beNot the general ledger. Not transaction detail. Not the QB import screen. F-BK1a triages unreconciled items by required attention — drill to transaction detail for line items · journal entry composition stays in QB until V1.5 bidirectional sync lands.
