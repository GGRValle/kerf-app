On notifications: the V1 phase uses Slack as one approval channel because Christian is sole approver and lives there — but Slack chrome is not the Kerf UI. Channel ≠ UI

  • The notification on the left is rendered in iOS lock-screen style for illustration. Its actual delivery channel is whatever the operator configured: Slack push, SMS, iMessage, email, OS push. Per project_kerf_decision_packet.md: "messaging is channel-agnostic; Slack is one V1 phase channel."
  • The right panel is the Kerf mobile UI the user lands in after tapping the notification. Canon mobile v2 fidelity. This is what Kerf's app looks like.
  • Slack chrome (#kerf-decisions sidebar, channel header, message threading) is never the Kerf UI. A Slack-rendered version of the Decision Packet exists in V1 because Christian is sole approver and approves from inside Slack — but the v0_2 Flow A frames that depict Slack as the UI design are corrected here in favor of channel-agnostic notification + Kerf-native UI.

v0.2 §5.1 Decision Packet schema rendered in Kerf UI

  • question, recommendation, recommendation_reason with 4 source items
  • evidence_ids[] with source-layer chips per item (TENANT_MEMORY · PUBLIC_REFERENCE)
  • artifact_effect, memory_effect, lifecycle transition (effects-if-approved block)
  • Three primary actions: Defer / Edit / Approve. Same contract as every Decision Packet (per project_kerf_decision_packet.md §5.2)

What the operator sees

  • Notification → tap → app opens to this exact Decision Packet detail. No "go to channel, find message, scroll" friction.
  • If the notification routes via Slack, tapping the Slack push uses Slack's deep-link to open Kerf — not navigate inside Slack.
  • If channel is SMS, tapping the SMS link opens Kerf to the same packet.
