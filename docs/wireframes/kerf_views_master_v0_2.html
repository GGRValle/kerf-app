Three actions (v0.2 §5.2 contract)

  • Approve & send — primary green button. Lifecycle: READY_FOR_REVIEW → APPROVED. Triggers artifact_effect (CO PDF + e-sign link). Slack confirms in thread.
  • Edit — opens Slack modal with editable fields. Operator's edits create a Layer C ESTIMATE_OVERRIDE scoped to this decision only. Logged as LearningSignal but not promoted to TENANT_MEMORY.
  • Reject — opens Slack modal asking for category + free-text reason. Generates a LearningSignal event. Right Hand never silently retries.
  • Defer 1h — secondary affordance. Status stays READY_FOR_REVIEW. Card re-surfaces in 1h.

  • If any chip were MODEL_INFERENCE, the action button would default to "Edit" instead of "Approve" — inference can never auto-approve.

What "Edit" does (v0.2 §5.2)

  • Creates an ESTIMATE_OVERRIDE event in the Blackboard. Scope: this Decision Packet only.
  • Bumps the labor adjustment from $4 to $5. Not silently promoted to TENANT_MEMORY.

Why two-step promotion (architectural firewall)

  • Pattern detection (3+ similar overrides over 30 days) triggers the weekly LearningSignal review surface — operator confirms intent across cases, then promotes.

What "Reject" produces (v0.2 §5.2)

  • A LearningSignal Blackboard event. Required: category (one of 6 enums). Optional: what_i_saw_that_you_missed + what_to_do_differently free-text.
  • Decision Packet status: READY_FOR_REVIEW → REJECTED. No artifact generated. No Platform write.
  • Right Hand never silently retries this exact decision. The original decision_id is closed.
  • Right Hand does NOT re-train on the rejection in real-time. The signal queues for weekly Auditor review (per Architecture v3.4 §26.10).

Why the two free-text fields are separate

  • "What did you see that I missed" — operator's mental model. Used for evidence detection logic, not behavior change.
  • "What should I do differently" — operator's prescription. Used for prompt-tuning / pattern-matching in next-cycle drafts.
  • Mixing them would let the operator's prescription overfit Right Hand to one case. Keeping them separate lets the weekly review weigh patterns across multiple LearningSignals before any prompt change.

Operator-trust property

  • The reject path is intentionally heavier than approve. Approve is one tap; reject asks for a category + reason.
  • This asymmetry encodes a value: rejecting Right Hand is informative. The system gets smarter from rejections; therefore rejections deserve a few extra seconds.
  • Operator never feels punished for rejecting. The modal copy says "What did you see that I missed?" not "Why are you wrong?".
